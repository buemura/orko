# @synkro/core v0.9.0

**Release Date:** 2026-03-06

## Breaking Changes

### `logger.warn` now always emits (TD-01)

`logger.warn` was previously gated behind the `debug: true` flag, meaning operational warnings (handler retry failures, workflow step mismatches, missing chained workflows) were silently suppressed in production. This made it difficult to diagnose issues without explicitly enabling debug mode.

**Before:** Warnings were only visible when `debug: true` was set in `SynkroOptions`.

**After:** Warnings always emit via `console.warn`, regardless of the `debug` flag â€” matching the existing `logger.error` behavior. The `debug` flag now exclusively controls verbose debug traces (`logger.debug`).

**Migration:** If you relied on warnings being suppressed (e.g., in test output), you may see new `console.warn` output. No code changes are required.

## Bug Fixes

### Multiple subscriptions no longer silently overwrite callbacks (TD-02)

`RedisManager` stored a single callback per channel, so registering multiple subscriptions to the same channel would silently discard previous callbacks. Similarly, `HandlerRegistry` stored a single handler per event type, overwriting any previously registered handler.

Both now use `Set`-based storage and fan out to all registered callbacks/handlers, matching the existing `InMemoryManager` behavior.

**Before:** Only the last registered callback/handler for a given channel/event would execute.

**After:** All registered callbacks/handlers execute for every message. `RedisManager` also avoids redundant Redis `SUBSCRIBE` calls when multiple callbacks target the same channel.

### Transport-level message deduplication (TD-03)

Redis Pub/Sub was delivering duplicate messages to the subscriber, causing cascading noise in debug logs â€” duplicate handler executions, duplicate workflow transition checks, and redundant Redis roundtrips for dedup verification. Each workflow step amplified the problem, with later steps receiving progressively more duplicates.

`RedisManager` now deduplicates messages at the transport layer before invoking any callbacks, using a bounded in-memory cache keyed by `channel + requestId`. This eliminates duplicates at the source, before they reach `HandlerRegistry` or `WorkflowRegistry`.

**Key details:**
- Extracts `requestId` from the message via regex (no full JSON parse) for dedup keying
- Falls back to full message content for messages without `requestId`
- Bounded cache (max 10,000 entries, 5-second window) with automatic eviction to prevent memory leaks
- Zero Redis roundtrips â€” purely local O(1) Map lookup

**Before:** Dozens of duplicate log lines per workflow execution (`Duplicate transition ignored`, `in-flight message ignored`, `duplicate message ignored`, redundant `[RedisManager] message on` lines).

**After:** Each channel receives exactly one message per `requestId`. Downstream dedup mechanisms (`HandlerRegistry`, `WorkflowRegistry`) are no longer triggered by transport-level duplicates.
